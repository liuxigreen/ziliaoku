# 咱们流水线 vs 竞品 7 步 —— 对照与完善

> 核心判断：**他的 7 步是「账号运营闭环」（怎么把一个号从 0 跑起来）；咱们的 9 段是「内容生产流水线」（怎么系统地产出好内容）。两者互补，不是替代。** 冲着"写文章涨粉"这个目标，咱们的**生产引擎已经很能打**，但**缺一层"平台适配 + 运营闭环"**。

## 一、逐条对照

| 竞品 7 步 | 咱们对应 | 谁强 |
|---|---|---|
| 1 先定变现方式 | `account.md` 静态配置（变现路径） | 他：**显式前置**、且"内容方向由变现倒推"；咱们只当配置，没驱动选题 |
| 2 找低粉爆款拆公式 | `titles_pool`（仅标题） | **他强**：主动拆目标平台自家爆款的"结构/开头/标签"；咱们只抽标题，不拆结构 |
| 3 建账号档案 | `account.md` | 平手 |
| 4 建选题库 | `topics` 选题库（20 个，带 evidence/fit_score） | **咱们强**：结构化、有证据、有评分 |
| 5 选题+生成内容 | `draft` 成稿（含去AI味） | **咱们强**：去AI味是硬约束；他只是"用范本改写" |
| 6 去AI味+合规检查 | `gate` 红线 + `draft` 去AI味 | **咱们强**（去AI味）；但**缺"发布前合规自检关卡"** |
| 7 排期+每周复盘 | `review`（含小眼睛/点赞/收藏/评论/涨粉闭环） | **咱们强**：数据反哺 formulas/image-styles 权重 |

## 二、咱们已经强在哪（保留，别动）

- **采集广度**：Reddit/YouTube/GitHub/公众号/热榜多源，他只盯小红书自身。
- **质检 gate**：自动过滤软广/垃圾，他不筛。
- **聚类 cluster + 风向标 signal**：跨源主题聚合 + 趋势感知，他没有。
- **配图 image**：系统出图，他只说"做封面"。
- **去AI味 + 复盘数据闭环**：两条都是硬约束/自动反哺，他不系统。

## 三、针对"涨粉"目标的 3 个真缺口

### 缺口 A（最大）：没有"拆平台爆款"环节
他第 2 步的核心是**研究目标平台自家正在跑赢的笔记**（低粉高赞 → 拆标题/开头3句/结构/标签），这是涨粉的命门——**知道写什么不如知道"在这个平台什么写法能火"**。
咱们现状：小红书只当"标题信号源"（抽标题进 `titles_pool`），**从不结构化解构 winning post**；公众号更只到发现层（抓全文还被验证码挡）。结果：咱们知道"选题好不好"，不知道"在小红书/公众号这个写法灵不灵"。
→ **完善**：把 `collect` 的小红书/公众号信号源从"只抽标题"升级为"结构化解构"（标题 + 开头3句 + 骨架 + 标签 + 数据），产物进 `data/xhs_winning_structures.jsonl`（与 titles_pool 并列），喂给 `draft` 当结构参考。

### 缺口 B：选题没显式回扣变现路径
他第 1 步"内容方向由变现倒推"是贯穿全流程的。咱们 `topics` 只要求"贴合人设与红线"，**没让每条选题标明它服务变现的哪一环**（引流型 / 信任型 / 转化型）。
→ **完善**：`topics` 输出每项加 `monetization_role: 引流|信任|转化`，保证每周选题组合能推动"小红书→私域→订阅"漏斗，而不是散装内容。

### 缺口 C：缺"发布前合规自检" + "内容排期"
- 他第 6 步有独立"合规检查 Prompt"（扫绝对化/收益承诺/硬引流）；咱们红线散在 `account.md`+`draft` 规则里，**没有一道发布前的自动红线扫描关卡**。
- 他第 7 步有"30天排期表"（养号期/标签识别期/稳定变现期 + 每日发布时间/类型/公式）；咱们产出选题库但**没有发布日历/节奏**。
→ **完善**：
  - `gate` 加 `publish_check` 模式：输入成稿 draft → 输出红线问题清单（绝对化词/收益数字/硬引流），发布前必经。
  - `topics` 或新文档产出 `周排期`（哪天发/发什么类型/套用哪条公式），保证稳定节奏（算法偏爱稳定更新）。

## 四、一个文档缺口（非流水线）

### 缺口 D：养号 / 冷启动运营 SOP
他额外必做：养号 3-5 天、两个手机、账号名+简介参考爆款。这是**纯平台运营**，不在内容生产流水线内，但涨粉前置动作。
→ **完善**：单独立 `docs/account_ops_sop.md`（养号/简介/发布节奏/双机），不进 9 段接口，作为"发布前 checklist"。

## 五、一个 refinement：标题公式分平台适配
`titles_pool` 现在小红书/公众号/热榜混在一起。但两平台标题风格迥异（小红书：短钩子+emoji+痛点；公众号：长悬念+干货感+数字）。
→ **完善**：`titles_pool` 每条加 `platform` 字段已有了，但**生成时按目标平台筛选套用**；`draft` 按发布平台选公式，不混用。

## 六、完善方案（不破坏冻结接口）

| 完善项 | 落点 | 是否破坏冻结接口 |
|---|---|---|
| A 拆平台爆款 | 扩展 `collect` 小红书/公众号信号源 → 结构化解构；新产物 `data/xhs_winning_structures.jsonl` | 否（信号源内部增强，输入/输出契约不变） |
| B 选题回扣变现 | `topics` 输出加 `monetization_role` 字段 | 否（仅扩展输出字段） |
| C1 发布前合规自检 | `gate` 加 `publish_check` 模式 | 否（gate 新增一个子模式） |
| C2 内容排期 | 新 `docs/publish_calendar.md` 或由 `topics` 顺带输出排期建议 | 否 |
| D 养号 SOP | 新 `docs/account_ops_sop.md` | 否（独立文档） |
| E 标题分平台 | `draft` 按 `platform` 选公式 | 否 |

> 设计纪律（入口≤6、发现抓取分离、人工终审三卡点）全部不动。新增的都是"能力扩展"而非"改既有接口"。

## 七、建议落地顺序

1. **先补 A（拆平台爆款）**——对涨粉杠杆最大，且直接喂 `draft` 结构质量。
2. **再做 C1（发布前合规自检）**——守住红线，防限流（限流=掉粉）。
3. **B + C2（选题回扣变现 + 排期）**——让内容有节奏、有漏斗。
4. **D（养号 SOP）**——发布前的运营底子。
5. **E（标题分平台）**——顺手做，边际成本低。
