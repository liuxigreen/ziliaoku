# 流水线对比审计：ziliaoku vs chenjin-cmd/agent-skills-launch-pack_

> 日期：2026-07-08
> 对比对象：`chenjin-cmd/agent-skills-launch-pack_`（225★/31fork，2026-06-13 建，7-8 更新）
> 用户目的：**收集信息 → 打信息差 → 写文章 → 发各平台，工具类/操作类/AI 优先，运营自己社交账号**
> 守纪律：本报告所有优化建议均"不破坏冻结接口"（仅扩输出字段 / 加子模式 / 独立文档），与 2026-07-08 竞品审计（@Eejoylove 7步）同源。

---

## 0. 一句话结论

**两个系统是互补的，不是替代关系。**
- 他们 = **按平台分的"起号规划包"**（5 平台 × 定位/选题库/30天日历/复盘/合规替代表），强在多平台运营方法论 + 工程化（CI/校验/单测/安装），**但不采集任何外部信息**。
- 我们 = **按流水线阶段分的"信息差采集→写作"流水线**（collect→gate→extract→cluster→signal→topics→draft→image→review + decode），强在真实采集 + 工具拆解 + 爆款拆解 + 数据流自动化，**但缺运营规划深度**。

用户目的是"收集信息打信息差写文章发各平台"——**前半句（收集+写）我们强、他们弱；后半句（发各平台运营）他们强、我们弱**。融合方向 = 保留我们采集+写作优势，补他们的运营规划方法论。

## 1. 根本差异：切法不同

| 维度 | 他们（agent-skills-launch-pack_） | 我们（ziliaoku） |
|---|---|---|
| 切法 | **按平台**分 skill（公众号/小红书/抖音/视频号/X 各一个起号专家） | **按流水线阶段**分 skill（9 段 + decode 横切） |
| 形态 | 规划框架（定位/选题/日历/复盘），纯 LLM 出计划 | 数据流（raw→gate→extract→...→review），有真实抓取与中间产物 |
| 采集 | ❌ 无（不收集外部信息，只规划） | ✅ collect→extract→decode 真实抓 GitHub/Reddit/X/小红书信号 |
| 工具拆解 | ❌ 无 | ✅ extract 拆工具 README（用户要的"工具类"强项） |
| 爆款拆解 | ❌ 无 | ✅ decode 拆写法进 winning_cases |
| 多平台发布 | ✅ 5 平台起号 skill | ⚠️ 仅小红书（且半自动，opencli files bug） |
| 工程化 | ✅ validate_skills.py + unittest + GitHub Actions CI + install.sh | ❌ 无校验/测试/CI/安装 |
| 合规 | ✅ 合规替代表（给替换词）+ 平台规则需核验 | ⚠️ publish_check 五类红线（只报错，没给替换词） |

**关键洞察**：他们解决"怎么把账号起来"，我们解决"怎么持续产出信息差内容"。用户目的两者都要，但**采集+写作是地基**（没内容谈不上运营），所以我们的方向没错，补运营层即可。

## 3. 他们的亮点（按用户目的契合度排序，可学）

### P0 — 强烈建议补（投入小、价值大、不破坏接口）

1. **review 升级"失败环节分层诊断"**
   - 现状：最差1条三归类（选题/执行/门面）太粗。
   - 补法：review 输出加"失败环节漏斗"——曝光差(标题/封面) / 点击好但阅读差(开头钩子) / 阅读好但收藏差(获得感不足) / 收藏好但评论差(没留互动钩子) / 评论好但关注差(没立人设/没引导) / 关注好但转化差(资源型不够)。每环节给不同治法。
   - 接口：只扩 review 输出字段，不动数据流。

2. **gate publish_check 加"合规替代表"**
   - 现状：publish_check 抓到"最/第一"只报 error。
   - 补法：输出加 `replacements` 字段，每个命中词给 2-3 个替换方案（如"最实用"→"很实用/实测好用/落地"）。
   - 接口：只扩 gate 输出，不动 verdict 四态。

3. **draft 加"每篇先写8-12标题再选1"**
   - 现状：每篇 2 个候选 + 开头多样性铁律。
   - 补法：draft 子流程加"标题批量产"——先套 formulas.md 公式产 8-12 个标题，再用"开头多样性铁律+近5篇去重"筛 1。
   - 接口：draft 加子模式，不动输出契约。

4. **account.md 升级"定位句模板 + 门面元素系统"**
   - 现状：account.md 有人设+红线+变现，但没定位句、没转门面。
   - 补法：加"定位句"（我帮助[目标人群]用[AI Agent工作流]解决[内容产出痛点]获得[持续选题库]）→ 拆成昵称/简介/固定开场白/内容支柱/关键词宇宙。
   - 接口：独立文档，不动技能。

### P1 — 建议补（中等投入）

5. **topics 选题四维打分**
   - 现状：fit_score 单维。
   - 补法：fit_score 拆四维——喜欢(人设契合)/擅长(能写深)/难复制(护城河)/风口上升(时效)，每维 1-5 分，总分加权。
   - 接口：只扩 topics 输出字段。

6. **topics 补"30天冷启动阶段剧本"**
   - 现状：周排期 ongoing。
   - 补法：docs/publish_calendar.md 加"冷启动30天剧本"——定位期(1-3)→对标期(4-7)→首轮发布(8-14)→模式放大(15-21)→复盘聚焦(22-30)，每阶段发布数/类型/目标。
   - 接口：独立文档，topics 排期引用。

7. **补"9条内容实验测款"方法论**
   - 现状：写完即发。
   - 补法：docs/content_experiment.md——同定位下9角度（痛点教程/失败复盘/公开挑战/案例拆解/反常识/清单避坑/评论答疑/场景实验/系列预告），每条只改1-2变量，找高于账号中位数的相对爆点再拆解进 winning_cases。
   - 接口：独立方法论文档，draft 可选引用。

### P2 — 可选（看是否做多平台/工程化）

8. **多平台发布 skill**（用户要"发各平台"）
   - 现状：仅小红书半自动。
   - 补法：若做多平台，按需加公众号/视频号/X 发布 skill（守"小红书主战场"纪律，其他平台按需）。
   - 注意：opencli 支持 twitter/xiaohongshu/bilibili，公众号走搜狗微信+半自动。视频号无 opencli 适配，暂缓。

9. **工程化：validate_skills.py + CI**
   - 现状：10 个 skill 无校验。
   - 补法：仿他们的校验器——结构(每个 skill 有 SKILL.md) + frontmatter(name/description) + 敏感内容扫描(本机路径/API key/cookie/token) + GitHub Actions CI。
   - 价值：skills 多了该有，防泄密（我们 mcp.json 有 firecrawl Key，虽已 gitignore，校验器兜底）。

## 4. 我们的优势（他们没有，不可丢）

1. **真实采集层**：collect 5入口 + extract 拆工具README + decode 拆写法——用户目的"收集信息打信息差"的核心，他们完全没有。
2. **工具/GitHub 拆解**：extract 拆 55 篇工具 README，用户要的"工具类/操作类"强项，他们没有。
3. **爆款拆解 decode + winning_cases 写作库**：从爆款学写爆款，draft 引用而非死套模板，他们没有。
4. **数据流自动化**：raw→gate→extract→cluster→topics→draft→image→review 冻结契约 + 中间产物，他们是一堆散装规划 skill 无数据流。
5. **去AI味/不画饼/开头多样性纪律**：系统化写作纪律，他们只有"朋友脑改写"单点。
6. **review 联动上游**：discard率→调初筛词、collect率→入口质量，复盘反哺采集层（他们不采集，无此闭环）。
7. **firecrawl 额度纪律**：opencli 免费优先、firecrawl 付费兜底（他们无此问题因不采集）。

---

## 5. 不建议照搬

1. **按平台分 skill**：他们是给规划师手动用的，按平台分合理；我们是自动化数据流，按流水线阶段分更适合。强行按平台分会破坏数据流。
2. **30天日历的"养号期/标签识别期"硬阶段**：我们已有周排期 + 养号期≤3篇/周纪律，够用；硬套阶段论反而僵化。
3. **他们的 frontmatter 极简（仅 name/description）**：我们 frontmatter 有 agent_created 等字段，更全，不退化。
4. **他们的"合规替代表"按平台**：我们 publish_check 跨平台五类红线已够，替代表按"命中词给替换"即可，不必每平台一份。
