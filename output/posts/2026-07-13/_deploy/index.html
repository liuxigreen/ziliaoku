用 Claude Code 大半年，中间一度觉得自己用得挺溜。直到刷到那期「32 个技巧，16 分钟带你从小白到量产工作流」，才意识到自己一直在裸奔——最该开的开关、最该养成的习惯，一个都没碰。

那期作者把技巧分成新手、进阶、高手三档。我结合自己这半年的踩坑，把 32 招重新归类，留下真正能当天用、且不容易踩坑的部分，按「地基 → 上下文 → 协作 → 规模化」四块讲。

▲ 32 招四块分类：地基 · 上下文 · 协作 · 规模化

01

PART


地基：两招吃透，已赢过大多数

FOUNDATION · 新手先打这两下

SKILL 1

进任何项目，先打 /init

/init 会让 Claude Code 扫一遍代码库，自动生成一份 CLAUDE.md——相当于这个项目的「速查卡」，把架构、规范、关键文件都记下来。以后每次新会话，它不用你重新解释背景，直接带着上下文上手。空仓库也能让它帮你建第一版。

我的理解：以前每个新会话都要把项目背景重说一遍，又累又容易跑偏。/init 等于让 AI 先读说明书再干活，成本几秒钟，回报是之后每次都少解释、少跑偏。

SKILL 2

永远先 Plan Mode（Shift+Tab 切换）

规划模式下它能读代码、能调研、能列方案，但一个字都不会改。等方案磨清楚，你再切回自动执行。更高阶的玩法是把 AI 当 junior 开发：别总下命令「写个 X 函数」，试着给它问题——「怎么处理增长追踪？」让它自己想方案。

我的理解：我实测过，90% 的返工都是「一上来就写」导致的。多花两分钟说清要干啥，比写完发现方向错、再改五遍快得多。

02

PART


上下文：AI 变傻，八成是喂涝了

CONTEXT · 噪音越少越好

3. 保持上下文小：别把整个代码库倒进一个对话，只给当前任务需要的，把大问题拆成小的、聚焦的步骤。噪音越少，它表现越好。

4. 用 /context 找 token 黑洞：它会把「谁在吃你的 token」拆成百分比——系统提示、文件内容、MCP 服务，一目了然，会话臃肿时就能定位问题。

5. 60% 就 /compact，换任务就 /clear：上下文到 60% 左右敲 /compact 压缩历史、重要信息不丢；完全换任务就用 /clear 彻底清空——好在 CLAUDE.md 还在，不是真从零来。

我的理解：上下文腐烂是很多人天天在发生的坑。AI 不是突然变笨，是你喂太多无关信息把它淹了。/context 是体检，compact 是瘦身。

03

PART


协作姿势：让返工少一半的三招

COLLAB · 对齐与验证

6. 让它主动提问，别闷头干：直接下指令「直到你有 95% 确信你懂我要什么，再动手」，它会用提问工具反复和你对齐，省掉来回改三四个回合。

7. 把「自检」写进待办清单：它每列一个 to-do，你紧跟一条验证——截图看排版、开 DevTools 看报错、不到 95% 确信不许进下一条。质量检查直接烤进执行计划。

8. 用子代理并行，主线程不卡死：复杂问题派子代理，每个有独立上下文、可跑不同模型，同时调研、写测试、试方案。简单活儿子代理用 Haiku、主线程用 Opus，又快又省钱。

我的理解：第 6、7 两招说的是同一件事的两边——对齐和验证。官方原话是「如果你只学一招，就学验证」，ROI 最高，没有之一。

04

PART


规模化：把重复劳动变成资产

SCALE · 攒肌肉记忆

9. 把重复的事存成自定义技能：在 .claude/skills 里建可复用提示词文件，比如教它扫技术债、审代码。下次自然语言调一句就行，不用每次重写。一周做超过两次的活儿，就值得存成技能。

10. 给它建一本「错题本」：只要 AI 做错一次，就把教训写进 CLAUDE.md 并提交。CLAUDE.md 不该是一次性说明书，而是越用越厚的错题本——不是教 AI，是和 AI 一起长记性。
