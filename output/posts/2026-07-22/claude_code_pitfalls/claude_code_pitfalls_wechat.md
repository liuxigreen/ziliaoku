# 我用 Claude Code 半年，替你踩了这 3 个坑

## 一、先说结论：Agent 不是甩手掌柜

半年里我用 Claude Code（以下简称 CC）重写过一个不小的项目，也拿它跑内容生产的自动化流水线。踩的坑，比看过的教程都多。

网上到处是"CC 真香""一天出一个应用"的成功学，但没人跟你讲翻车。今天我反过来，把真金白银（主要是时间和头发）换来的三个坑摊开讲——每一个都让我在某个深夜怀疑人生。如果你也在用 CC，希望你能绕开。

一个前提：我不是来劝你别用的。CC 现在是我的主力，离开它我效率直接腰斩。问题从来不在工具，在我的用法。

## 二、坑一：把它当黑盒，甩句需求就等结果

最早我干过最蠢的事：丢一句"帮我做个 XX 功能"，然后就去摸鱼了，心想要不了一会儿就好。

回来一看，它自作主张改了一堆不相干的文件，方向还偏了——本来要加个按钮，它顺手把我的目录结构也"优化"了，还把两个老接口给删了。

为什么会这样？因为人一旦把"结果"甩出去，就容易进入"它应该懂我"的幻觉。CC 不会读心，它只会按你给的最稀薄的信息，往最省力的方向跑。

代价很直接：返工比我自己写还累，而且信任一旦被透支，后面你就不敢放手了，又退回纯手写，白用。

**怎么避**：别甩黑盒。先给它"方向 + 约束 + 验收标准"，让它先用 plan 模式列出步骤，你确认了再动手。具体点：让它先输出 3-5 步计划，标出会动哪些文件、动什么，我点头才跑。这一步多花两分钟，省下的是两小时返工。CC 强在"跟你一起干"，不是"替你干"。把决策权握在自己手里，它才是加速器。一句话：你当项目经理，它当执行。

## 三、坑二：不写开发文档，上下文一丢就失忆

第二个坑更隐蔽。大项目跑个几天，会话一清，CC 把之前定好的架构、踩过的雷全忘了。同一个低级错误它能连犯三遍，你还以为是新 bug，又花两小时查。

最坑的是：它看起来"很忙很有产出"，实际在重复踩雷。进度是假象，埋雷是真事。

我后来看到一个 Reddit 上单人跑通 30 万行项目的大佬，他的核心招之一就是"dev docs workflow that prevents Claude from losing the plot"——用一份开发文档，防止 CC 跑偏。我照做之后才反应过来：这份文档主要不是给人看的，是给 Agent 当锚的。

**怎么避**：养一份 dev docs，内容很轻量——架构决策（为什么这么分）、关键约束（哪些不能动）、已踩的坑（别再试）。不用长篇大论，十行就够，重点是"哪些不能动"和"为什么"。每次开新会话先让它读这份文档。CC 一旦有了"记忆"，失忆式翻车就基本消失了。

## 四、坑三：全自动一把梭，报错雪崩

第三个坑最痛。有过一次上头：让它一口气重构全站，想着"反正是它干，我睡一觉就好"。

结果半夜被报错叫醒，连环炸，我一个人手动救火到天亮，最惨的一次直接通宵。问题不在 CC 能力，在于我取消了所有刹车。没有检查点，错误就层层叠加，最后根本收不了尾，只能推倒重来。

那位大佬的解法很实在：用 PM2 + hooks 保证"zero errors left behind"（不留一个错误过夜），再配一支"专职 agent 部队"分工做评审、测试、规划。说白了，全是给失控加刹车。

**怎么避**：拆小步 + 用 hooks / CI 卡错误 + 每步留检查点。比如提交前自动跑类型检查，红色报错立刻拦下，不让错误混进历史。别追求"一次跑完"，追求"每一步都能停、能回溯"。小步快跑，永远比一把梭稳。失控不可怕，怕的是失控了还没刹车。

## 五、底层逻辑：Agent 是加速器，不是脑子

三个坑拆开看，根因是同一个——我把"思考、所有权、验收"这三件事外包了。

让它想方向（思考外包）、让它定结构（所有权外包）、让它判对错（验收外包），那它当然会跑偏，因为没人掌舵。CC 越强，越要你掌舵。它替你跑腿、替你搬砖，但方向得你定、上下文得你记、质量得你卡。

把 Agent 当同事，别当许愿池。同事你会给 briefing、会 review、会兜底；许愿池你只会等奇迹。

## 六、落地原则（可直接抄）

1. 先计划后执行，绝不甩黑盒——你当 PM，它当执行。
2. 文档当 Agent 的锚，不是给人看的摆设——失忆式翻车靠它灭。
3. 小步快跑 + 检查点，拒绝一把梭——每一步都能停、能回溯。
4. 关键模块自己把关，重复活交给 AI——脑子留给你，搬砖给它。

这四条我现在天天用，踩坑频率降了一大半。CC 是好工具，但好工具也挑人——挑的是"愿不愿意自己先想清楚"的人。

如果你现在就在用 CC，今天就能试一件小事：给当前项目建一份一页纸的 dev docs，下次开会话先让它读。三天后回来告诉我，翻车是不是少了。
